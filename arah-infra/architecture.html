Infrastructure Architecture

AWS ap-southeast-1 · Terraform · EKS · ElastiCache

AWS Architecture Overview

All production workloads run in AWS ap-southeast-1 (Singapore). Public traffic enters via CloudFront CDN, passes through an Application Load Balancer (nginx ingress), and is routed to five EKS-hosted services.

                    ┌─────────────────────────────────────────────┐
                    │           AWS ap-southeast-1                 │

Terraform Module Breakdown

Seven reusable modules compose the full infrastructure stack. Each module is independently versioned and can be applied in isolation.

Module Resources Description
terraform/modules/vpc VPC, Subnets, NAT GW, Route Tables Network foundation with public/private subnets across 3 AZs
terraform/modules/eks EKS Cluster, Node Groups, OIDC Managed Kubernetes cluster with IRSA-ready OIDC provider
terraform/modules/redis ElastiCache Redis 7.1, Subnet Group Session store and routing cache for sub-50ms reads
terraform/modules/ecr ECR Repositories × 8 Private container registries with image scanning enabled
terraform/modules/cdn CloudFront + S3 Global CDN for map tiles and static assets
terraform/modules/dns Route53 zones, records DNS management with health-check failover
terraform/modules/monitoring kube-prometheus-stack, Grafana Full observability stack with pre-built dashboards

Kustomize Overlay Structure

Shared base manifests are patched per environment. Staging runs lighter replicas with debug logging; prod enables HPA, PodDisruptionBudgets, and stricter resource limits.

k8s/
├── base/               # Shared manifests (Deployments, Services, ConfigMaps)
│   ├── api-gateway/
│   ├── tile-server/
│   ├── valhalla/
│   ├── nominatim/
│   └── geocoding/
├── overlays/
│   ├── staging/        # Staging patches: 1 replica, debug logging, small limits
│   │   ├── kustomization.yaml
│   │   └── patches/
│   └── prod/           # Prod patches: 2+ replicas, HPA, PDB, prod resource limits
│       ├── kustomization.yaml
│       └── patches/

Kubernetes Workloads

All workloads run on EKS managed node groups with auto-scaling. HPA is enabled for stateless services; stateful services use fixed replicas.

Name Replicas (prod) HPA Resources
api-gateway 2–5 ✓ 200m CPU / 256Mi
tile-server 2–4 ✓ 500m CPU / 1Gi
valhalla 2 — 1000m CPU / 4Gi
nominatim 1 — 500m CPU / 2Gi

Scalability Path

Infrastructure evolves with user growth. Each phase represents a step-change in capability without a full rewrite.

Phase Users Infrastructure
MVP 0 – 10k Docker Compose on a single EC2 instance
Growth 10k – 100k ECS Fargate + ElastiCache Redis + RDS
Scale 100k – 1M EKS + HPA + multi-AZ Redis + CloudFront
National 1M+ Multi-AZ + Kafka + read replicas + global CDN
