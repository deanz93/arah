Contributing

Contributing to arah-infra

Branch format, commit conventions, and the PR checklist every contributor follows.

Getting Started

1
Fork & Clone

Fork the repo on GitHub, then clone your fork locally.

2
Create a Branch

Use the naming convention below — tied to a story ID.

3
Open a PR

Run the pre-PR checklist and fill in the PR template.

Branch Naming

All branches must be tied to a story ID in the format INF-NNN.

# Format
feature/INF-NNN-short-description

# Examples
feature/INF-001-terraform-s3-backend
feature/INF-011-api-gateway-deployment
feature/INF-023-staging-auto-deploy

Branches that don't follow this convention will be rejected by the branch protection rules.

Commit Format

Follow Conventional Commits with the infra scope.

# Format
feat(infra): short description of what was done

# Examples
feat(infra): add S3 remote state backend module
fix(infra): correct EKS node group IAM policy
chore(infra): update Terraform to 1.9.0
docs(infra): add vpc module usage example
Prefix When to use
feat(infra):New Terraform resource, K8s manifest, or CI step
fix(infra):Bug fix in IaC, broken pipeline, misconfigured resource
chore(infra):Version bumps, dependency updates, formatting
docs(infra):README, runbook, or inline comment updates
refactor(infra):Module restructure, variable renaming, no behaviour change

Required Skills

Read the relevant skill doc before opening a PR in that area.

docs/terraform.md

Module structure, variable naming, output conventions, and state management rules.

docs/k8s.md

Kustomize overlay layout, resource naming, label taxonomy, and RBAC guidelines.

docs/pr.md

PR template, review expectations, approval requirements, and merge strategy.

Pre-PR Checklist

Every PR must pass these checks before requesting review. CI will enforce them, but run locally first to catch issues early.

terraform validate
terraform -chdir=terraform/modules/<module> validate
terraform fmt
terraform fmt -recursive terraform/
Checkov IaC scan
checkov -d terraform/ --framework terraform
kubectl dry-run
kubectl apply -k k8s/overlays/staging --dry-run=server
All new AWS resources are tagged

Every resource must include: Project = arah, Environment = staging|prod, ManagedBy = terraform.

No hardcoded secrets

Secrets must use AWS Secrets Manager or SSM Parameter Store. Never hardcode passwords, keys, or tokens in Terraform or manifests. CI runs git-secrets automatically.

Review Process

Staging (auto)
  • ▸ 1 approver required
  • ▸ All CI checks pass
  • ▸ Merges to main auto-deploy
Production (manual gate)
  • ▸ 2 approvers required
  • ▸ Terraform plan reviewed in PR
  • ▸ Manual GitHub Actions approval
